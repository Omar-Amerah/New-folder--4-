## Performance Optimization: Avoiding Array Re-allocation

When repeatedly iterating and mutating state to test hypothetical scenarios (e.g., in a grid search or cost estimation loop), avoid using the spread operator (`[...design, newPart]`) inside the loop, as this creates a full clone of the array on every iteration.

Instead, append the temporary element once (`design.push(tempPart)`), mutate it in place during the iteration, and `pop()` it when done. This avoids O(N) array allocation overhead within inner loops, and was demonstrated to significantly reduce execution time (e.g., from ~0.7ms to ~0.6ms on average over 200 iterations for the `estimatePartEffectiveCost` function).
Moving 'require()' calls to the top of files instead of keeping them inside hot loop functions is an important optimization to reduce execution overhead in this node.js environment. Removing destroyed and disconnected ships from the player.ships array prevents memory leaks. The player.ships array must always be kept in sync with the room.ships map to avoid dangling objects, especially on player disconnect/refresh.
- Frontend rendering optimizations in `public/src/game/renderer.js` use off-screen canvases (`document.createElement('canvas')`) cached in a size-limited `Map` (`moduleCache`) to improve performance. Helper rendering functions (like `drawWeaponBase` or `drawRoundSystem`) accept an explicit `context` parameter instead of exclusively relying on the global `ctx` from `dom.js`.
